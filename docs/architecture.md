# Моделирование системы

## Логическая модель данных

Диаграмма отражает весь комплекс сущностей, участвующих в процессе, включая меню с сезонным фактором, корзину, состав товаров, систему лояльности, возмодности оплаты и др.

![](diagrams/DataModel.png)

## Описание сущностей логической модели данных

### Основные сущности системы

#### 1. Клиент
**Атрибуты:**
- `номер телефона` (string) - уникальный идентификатор клиента
- `участник бонусной программы` (boolean) - флаг участия в программе лояльности

**Связи:**
- Формирует Корзину (0..1)
- Совершает Заказы (1..*)

#### 2. Меню
**Атрибуты:**
- `тип меню` (enum) - категория меню (основное, детское, вегетарианское и т.д.)
- `название категории` (string) - наименование раздела меню
- `изображение` (blob/url) - картинка категории
- `наименование товара` (string) - название позиции в меню
- `изображение товара` (blob/url) - фото товара

**Связи:**
- Содержит Товары (1..*)
- Рекурсивная связь для вложенных категорий

#### 3. Ингредиент
**Атрибуты:**
- `название` (string) - наименование ингредиента
- `изображение` (blob/url) - визуальное представление
- `стоимость` (decimal) - цена за единицу
- `К` (int) - калории
- `Б` (int) - белки
- `Ж` (int) - жиры
- `У` (int) - углеводы

**Связи:**
- Входит в состав КомбоБургеров (0..*)
- Рекурсивная связь для составных ингредиентов

#### 4. Товар
**Атрибуты:**
- `название` (string) - наименование товара
- `описание` (text) - детальное описание
- `цена` (decimal) - стоимость
- `изображение` (blob/url) - фото товара
- `категория` (string) - принадлежность к категории меню
- `К/Б/Ж/У` (int) - пищевая ценность

**Связи:**
- Принадлежит Меню (1)
- Может входить в Корзину (0..*)
- Может быть составным (рекурсивная связь)
- Связан с КомбоБургерами (0..*)

#### 5. КомбоБургер
**Атрибуты:**
- `название` (string) - имя комбо-набора
- `ингредиент` (string) - составные компоненты

**Связи:**
- Состоит из Ингредиентов (0..*)
- Может быть представлен как Товар (0..*)

### Сущности заказа и оплаты

#### 6. Заказ
**Атрибуты:**
- `номер` (string) - уникальный идентификатор заказа
- `статус` (enum) - текущее состояние заказа
- `дата` (datetime) - время создания
- `оплата` (enum) - статус оплаты
- `способ получения` (enum) - самовывоз/доставка
- `время получения` (datetime) - когда забрать/доставить

**Связи:**
- Создается Клиентом (1)
- Происходит из Корзины (1)

#### 7. Корзина
**Атрибуты:**
- `стоимость` (decimal) - общая сумма
- `товар` (reference) - ссылка на товары
- `количество` (int) - число позиций

**Связи:**
- Формируется Клиентом (0..1)
- Превращается в Заказ (1)
- Содержит Товары (1..*)

#### 8. Оплата
**Атрибуты:**
- `сумма` (decimal) - оплачиваемая сумма
- `способ Оплаты` (enum) - тип платежа
- `чек` (string) - номер/код чека

**Связи:**
- Абстрактная сущность для наследования
- Связана с Корзиной (1)

#### 9. Карта (наследник Оплаты)
**Атрибуты:**
- `данные карты` (encrypted) - зашифрованные реквизиты
- Дополнительно наследует атрибуты Оплаты

**Связи:**
- Обрабатывается через Терминал (1)

#### 10. Наличные (наследник Оплаты)
**Атрибуты:**
- Дополнительно наследует атрибуты Оплаты

**Связи:**
- Принимаются через Купюроприемник (1)

### Бонусная система

#### 11. Бонусы
**Атрибуты:**
- `номер телефона клиента` (string) - идентификатор
- `код для оплаты` (string) - ваучер/промокод
- `шлюз для оплаты` (reference) - система обработки
- `баллы для оплаты` (int) - количество списываемых баллов

**Связи:**
- Является способом Оплаты (наследование)
- Интегрируется через Шлюз (1)

#### 12. Шлюз
**Атрибуты:**
- `адрес` (string) - API endpoint
- `имя` (string) - название системы лояльности
- `бонусы` (reference) - обрабатываемые бонусы

### Технические сущности

#### 13. Терминал
**Атрибуты:**
- `название` (string) - идентификатор устройства
- `номер` (string) - серийный номер
- `банк` (string) - обслуживающая организация

#### 14. Купюроприемник
**Атрибуты:**
- `номер` (string) - идентификатор устройства
- `банкнота` (enum) - тип принимаемых купюр
- `количество` (int) - число купюр
- `номинал` (decimal) - достоинство купюр

### Архитектура системы

Архитектурная диаграмма отражаеи все компоненты системы. За границей контура системы находятся компоненты с которыми система интегрирована для осуществления функци оплаты, складского хранения, уведомления, работы в рамках лояльности и проч.

# Архитектурное обоснование и описание потоков данных

## Выбор архитектурного стиля: Микросервисы

### Обоснование выбора

1. **Масштабируемость компонентов**
   - Независимое масштабирование сервисов аутентификации, обработки заказов и аналитики
   - Оптимальное распределение ресурсов под нагрузку

2. **Гибкость технологического стека**
   - Возможность использовать разные технологии для разных сервисов
   - Постепенная модернизация компонентов

3. **Повышенная отказоустойчивость**
   - Изоляция сбоев в пределах отдельных сервисов
   - Индивидуальные стратегии восстановления

4. **Соответствие бизнес-доменам**
   - Четкое разделение по бизнес-функциям
   - Каждый сервис отвечает за конкретную бизнес-возможность

## Описание ключевых потоков данных

### 1. Поток аутентификации

**Участники:**
- Клиентское приложение
- API Gateway
- Сервис аутентификации (Auth Service)
- База данных пользователей

**Шаги выполнения:**
1. Клиент → API Gateway: POST /login с учетными данными
2. API Gateway → Auth Service: Передача credentials
3. Auth Service → DB: Верификация пользователя
4. DB → Auth Service: Возврат данных пользователя
5. Auth Service → API Gateway: JWT токен
6. API Gateway → Клиент: Токен доступа

**Особенности:**
- HTTPS-шифрование всех запросов
- Хранение паролей в хэшированном виде
- Временные JWT-токены с ограниченным сроком действия

### 2. Поток обработки заказа

**Участники:**
- Клиентское приложение
- API Gateway
- Сервис заказов (Order Service)
- Платежный сервис (Payment Service)
- Сервис уведомлений (Notification Service)

**Основной сценарий:**
1. Клиент → API Gateway: POST /orders (данные заказа)
2. API Gateway → Order Service: Создание заказа
3. Order Service → Payment Service: Инициация платежа
4. Payment Service → Order Service: Подтверждение оплаты
5. Order Service → Notification Service: Запрос на уведомление
6. Order Service → API Gateway: Подтверждение заказа
7. API Gateway → Клиент: Детали созданного заказа

**Альтернативные сценарии:**
- Отмена заказа
- Возврат платежа
- Обновление статуса доставки

### 3. Поток аналитики

**Участники:**
- Административная панель
- API Gateway
- Аналитический сервис (Analytics Service)
- Хранилище данных

**Процесс формирования отчета:**
1. Администратор → API Gateway: GET /analytics?params
2. API Gateway → Analytics Service: Запрос данных
3. Analytics Service → DB: Агрегация данных
4. DB → Analytics Service: Результаты выборки
5. Analytics Service → API Gateway: Сформированный отчет
6. API Gateway → Администратор: Визуализация данных

**Типы отчетов:**
- Динамика продаж
- ТОП товаров/услуг
- География заказов
- Финансовые метрики

### 4. Поток уведомлений

**Участники:**
- Сервисы-инициаторы (Order, Payment и др.)
- Сервис уведомлений (Notification Service)
- Внешние SMTP/SMS/Push-сервисы

**Процесс отправки:**
1. Сервис-инициатор → Notification Service: Событие
2. Notification Service:
   - Определяет тип уведомления
   - Формирует контент сообщения
   - Выбирает канал доставки
3. Notification Service → Внешние сервисы: Отправка
4. Notification Service: Логирование результата

**Типы уведомлений:**
- Регистрационные
- Статус заказов
- Маркетинговые
- Системные оповещения
